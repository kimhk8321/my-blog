## 프론트엔드와 백엔드란?

하나의 거대한 소프트웨어를 **프레젠테이션 계층**과 **데이터 액세스 계층**으로 관심사를 분리한 각각의 계층을 의미합니다.

- **프론트엔드**: 소프트웨어에서 사용자에게 **보여지는 부분**을 총칭.
- **백엔드**: 사용자에게 감춰져 있으면서, 요구되는 **비즈니스 로직을 처리**하는 부분을 총칭.

## 서버

**서버**란 사용자로부터 요청을 받고, 그에 대한 리소스를 반환하는 컴퓨터 시스템입니다.

### 웹 서버 (Web Server)

클라이언트 요청 시 **정적 페이지를 전용으로 처리**합니다. HTTP 기반으로 정적 콘텐츠는 바로 제공하고, 동적 요청은 WAS로 넘긴 뒤 그 결과를 전달합니다. (Apache HTTP Server, Nginx)

### WAS (Web Application Server)

DB 조회·로직 처리가 필요한 **동적 콘텐츠를 제공**하는 애플리케이션 서버입니다. 주로 DB 서버와 함께 동작하며 비즈니스 로직을 처리합니다. (Tomcat, Resin, Jeus)

## HTTP

HTTP는 웹 클라이언트와 웹 서버 간에 **요청과 응답을 주고받을 때 쓰는 통신 규약**입니다. URL로 HTTP 요청을 보내면, 브라우저가 응답을 파싱해 사용자에게 보여줍니다.

**작동 방식**

1. **Request**: Header에 주소(URL)·포트·메서드를 담고, Body에 정보를 담아 요청을 전송.
2. **처리**: 서버에서 URI와 메서드에 해당하는 로직이 실행.
3. **Response**: Header에 상태 코드, Body에 반환 값을 담아 사용자에게 전달.

**HTTP 메서드**: `GET`, `POST`, `PUT`, `DELETE` 등. Request Header에 담겨 전송되며, **같은 URL이라도 메서드에 따라 서버는 다른 로직**을 수행합니다.

## API

**API**는 "HTTP 요청이 어떤 응답을 반환할지에 대한 규약"입니다. 이 규약은 **프론트엔드 개발자와 백엔드 개발자 간의 인터페이스**로 작동합니다. 구성 요소는 다음과 같습니다.

- 주소값(엔드포인트)
- HTTP 메서드
- 조건값: Query String, HTTP Body, Path Variable

---

정리하면, 프론트엔드와 백엔드는 **관심사 분리**의 두 축이고, 그 둘을 잇는 약속이 **HTTP와 API**입니다. 프론트엔드 개발자도 이 통신 구조를 이해해야 백엔드와 효율적으로 협업할 수 있습니다.
