## 프로세스 vs 스레드

- **프로세스**: 실행 중인 프로그램. 독립된 메모리 공간(코드/데이터/힙/스택)을 가집니다. 서로 격리되어 한 프로세스가 죽어도 다른 프로세스에 영향이 적습니다.
- **스레드**: 프로세스 내의 실행 흐름. **코드·데이터·힙을 공유**하고 스택만 따로 가집니다. 생성·전환 비용이 프로세스보다 작지만, 공유 자원 때문에 동기화가 필요합니다.

```
[프로세스]
 ├ 코드 / 데이터 / 힙  ← 스레드들이 공유
 ├ 스레드1 (스택)
 └ 스레드2 (스택)
```

## 컨텍스트 스위칭

CPU가 실행 중인 프로세스/스레드를 바꿀 때, 현재 상태(레지스터, 프로그램 카운터 등)를 **PCB**에 저장하고 다음 것을 복원합니다. 이 전환에는 비용이 들기 때문에, 너무 잦으면 성능이 떨어집니다.

## 동시성 문제

여러 스레드가 **공유 자원**에 동시에 접근하면 문제가 생깁니다.

- **경쟁 상태(Race Condition)**: 실행 순서에 따라 결과가 달라지는 상태.
- **임계 구역(Critical Section)**: 공유 자원에 접근하는 코드 영역. 한 번에 하나의 스레드만 들어가야 합니다.

```
// 두 스레드가 동시에 count++ 를 수행하면
// 읽기→증가→쓰기 사이에 끼어들어 값이 유실될 수 있다
count = count + 1;
```

## 동기화 기법

- **뮤텍스(Mutex)**: 자물쇠. 임계 구역에 **하나의 스레드만** 들어가게 합니다(상호 배제). 잠근 스레드만 풀 수 있습니다.
- **세마포어(Semaphore)**: 카운터 기반. **N개**까지 동시에 허용합니다. (뮤텍스는 N=1인 특수한 경우로 볼 수 있음)

## 데드락 (Deadlock)

둘 이상의 스레드가 서로가 가진 자원을 기다리며 **영원히 멈추는** 상태입니다. 다음 4가지 조건이 모두 성립할 때 발생합니다.

1. **상호 배제**: 자원을 한 번에 하나만 사용
2. **점유와 대기**: 자원을 쥔 채 다른 자원을 기다림
3. **비선점**: 남의 자원을 강제로 뺏지 못함
4. **순환 대기**: 대기가 원형을 이룸

→ 이 중 하나라도 깨면 데드락을 예방할 수 있습니다(예: 자원 획득 순서를 정해 순환 대기 제거).

## 프론트엔드와의 연결

JavaScript는 **싱글 스레드**라 이런 락 문제에서 비교적 자유롭지만, 대신 [이벤트 루프](/posts/js-deep-dive-async)로 동시성을 다룹니다. 무거운 작업은 **Web Worker**(별도 스레드)로 분리해 메인 스레드를 막지 않게 합니다.

운영체제의 동시성 개념은 언어를 막론하고 통합니다. "**공유 자원에 동시에 접근하면 위험하고, 그래서 동기화가 필요하다**"는 원리는 Web Worker, 서버 멀티스레딩 등 어디서든 적용됩니다.
