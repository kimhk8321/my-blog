## 전체 흐름

브라우저가 화면을 그리는 과정은 크게 다음과 같습니다.

```
HTML 파싱 → DOM 트리
CSS 파싱  → CSSOM 트리      ┐
                            ├→ 렌더 트리 → 레이아웃(Reflow) → 페인트(Paint) → 합성(Composite)
JS 실행 (DOM/CSSOM 조작)     ┘
```

1. **HTML 파싱 → DOM 트리**: 서버가 보낸 HTML을 위에서부터 파싱해 **DOM 트리**를 만듭니다.
2. **CSS 파싱 → CSSOM 트리**: `link`/`style`의 CSS를 파싱해 **CSSOM 트리**를 만듭니다.
3. **렌더 트리**: DOM과 CSSOM을 결합해, **화면에 그려질 노드만** 모은 렌더 트리를 만듭니다. (`display: none`은 제외)
4. **레이아웃(Layout / Reflow)**: 각 노드의 위치와 크기를 계산합니다.
5. **페인트(Paint)**: 실제 픽셀(색·텍스트·이미지)을 그립니다.
6. **합성(Composite)**: 여러 레이어를 합쳐 최종 화면을 만듭니다.

## 파싱을 막는 리소스

- **CSS는 렌더링 차단 리소스**입니다. CSSOM이 완성되어야 렌더 트리를 만들 수 있으므로, CSS 로딩이 늦으면 화면이 늦게 그려집니다. → CSS는 `<head>`에서 빠르게 로드.
- **JavaScript는 파싱을 차단**할 수 있습니다. `<script>`를 만나면 HTML 파싱을 멈추고 JS를 실행합니다.
  - **`defer`**: HTML 파싱과 **병렬로 다운로드**하고, 파싱이 끝난 뒤 순서대로 실행. (권장)
  - **`async`**: 병렬 다운로드 후 **다운로드되는 즉시** 실행(순서 보장 안 됨).

```html
<script src="app.js" defer></script>
```

## 리플로우와 리페인트

- **리플로우(Reflow)**: 레이아웃에 영향을 주는 변경(크기·위치·DOM 추가/삭제). 위치·크기를 다시 계산하므로 비용이 큽니다.
- **리페인트(Repaint)**: 레이아웃은 그대로지만 색·배경 등 시각적 속성만 변경. 리플로우보다 저렴합니다.

### 성능 팁

- 레이아웃을 유발하는 속성(`width`, `top`, `left` 등) 대신, **`transform`·`opacity`** 로 애니메이션하면 리플로우 없이 합성 단계에서 처리되어 빠릅니다.
- DOM을 반복적으로 조작할 때는 **한 번에 모아서** 변경합니다(예: `DocumentFragment`).
- 레이아웃 값을 읽고(`offsetHeight`) 바로 쓰는 동작을 번갈아 하면 **강제 동기 레이아웃**이 발생하니 주의합니다.

## 타이머

```javascript
const id = setTimeout(() => console.log("1초 후"), 1000);
clearTimeout(id);

const iid = setInterval(() => console.log("매 1초"), 1000);
clearInterval(iid);
```

타이머의 콜백은 [(3)편의 이벤트 루프](/posts/js-deep-dive-async)에서 본 것처럼 **매크로태스크 큐**를 거쳐, 콜 스택이 빈 뒤에 실행됩니다. 따라서 지정한 시간은 "**최소 대기 시간**"이지 정확한 실행 시점이 아닙니다.

렌더링 과정을 이해하면, 왜 CSS는 위에 두고 JS는 `defer`를 쓰는지, 왜 `transform`으로 애니메이션하는지가 자연스럽게 설명됩니다. 성능 최적화의 출발점이 되는 지식입니다.
