객체지향 프로그래밍(OOP)은 데이터와 동작을 **객체**로 묶어 설계하는 패러다임입니다. 핵심 특성과, 좋은 설계를 위한 **SOLID 원칙**을 정리합니다.

## 객체지향 4대 특성

- **캡슐화(Encapsulation)**: 데이터와 메서드를 묶고, 내부 구현을 숨긴다(정보 은닉).
- **상속(Inheritance)**: 기존 클래스를 확장해 재사용한다.
- **다형성(Polymorphism)**: 같은 인터페이스로 다른 동작을 한다.
- **추상화(Abstraction)**: 복잡한 것을 본질만 남겨 단순하게 표현한다.

## SOLID 원칙

유지보수·확장에 강한 객체지향 설계를 위한 다섯 원칙입니다.

### S — 단일 책임 (Single Responsibility)

한 클래스(모듈)는 **하나의 책임**만 가져야 한다. 변경 이유가 하나여야 한다.

```ts
// 나쁨: 사용자 정보 + 저장 + 이메일 발송이 한 곳에
class User { save() {} sendEmail() {} }

// 좋음: 책임 분리
class User {}
class UserRepository { save(user: User) {} }
class EmailService { send(user: User) {} }
```

### O — 개방-폐쇄 (Open-Closed)

확장에는 **열려**, 수정에는 **닫혀** 있어야 한다. 새 기능 추가 시 기존 코드를 고치지 않는다.

```ts
// 새 도형이 생겨도 area 함수를 수정하지 않도록, 각자 area를 구현
interface Shape { area(): number; }
class Circle implements Shape { area() { return Math.PI * this.r ** 2; } }
class Square implements Shape { area() { return this.side ** 2; } }
```

### L — 리스코프 치환 (Liskov Substitution)

자식 타입은 부모 타입을 **대체해도** 프로그램이 정상 동작해야 한다. (예: `Square`가 `Rectangle`을 상속하면서 동작을 깨면 위반)

### I — 인터페이스 분리 (Interface Segregation)

쓰지 않는 메서드에 의존하지 않도록, **인터페이스를 작게** 나눈다.

```ts
// 나쁨: 거대한 인터페이스
interface Worker { work(): void; eat(): void; }

// 좋음: 필요한 것만
interface Workable { work(): void; }
interface Eatable { eat(): void; }
```

### D — 의존성 역전 (Dependency Inversion)

구체 구현이 아니라 **추상(인터페이스)에 의존**한다. 상위 모듈이 하위 모듈의 세부에 묶이지 않게 한다.

```ts
// 추상에 의존 → 구현(MySQL/Mongo)을 갈아끼워도 Service는 그대로
interface Repository { save(data: unknown): void; }
class Service {
  constructor(private repo: Repository) {} // 주입
}
```

## 프론트엔드에서의 의미

리액트는 함수형 색이 짙지만, SOLID의 사상은 그대로 통합니다 — **단일 책임**(작은 컴포넌트/훅), **개방-폐쇄**([합성](/posts/design-patterns)으로 확장), **의존성 역전**(props/context로 주입). 패러다임이 달라도 "**변경에 강한 구조**"라는 목표는 같습니다.

---

OOP와 [함수형](/posts/functional-programming)은 대립이 아니라 **도구상자의 다른 칸**입니다. SOLID는 특정 언어를 넘어 "어떻게 하면 바꾸기 쉬운 코드를 만들까"에 대한 검증된 지침입니다.
