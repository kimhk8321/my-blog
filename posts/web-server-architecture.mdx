웹 서비스가 어떤 구조로 동작하는지, DBMS에서 시작해 미들웨어·WAS·게이트웨이까지 발전 과정을 따라가며 정리합니다.

## DBMS와 미들웨어의 등장

**DBMS**(Database Management System)는 다수의 사용자가 데이터베이스에 접근하도록 해주는 소프트웨어입니다(MySQL, Oracle 등). 보통 서버 형태로 제공되어, 클라이언트가 직접 DBMS에 연결하는 방식이 쓰였습니다. 하지만 이 방식은 단점이 있었습니다.

- 클라이언트의 **로직이 많아지고 크기가 커진다.**
- 로직이 바뀌면 **클라이언트를 매번 재배포**해야 한다.
- 로직이 클라이언트에 포함되어 배포되므로 **보안이 나쁘다.**

이를 해결하기 위해 등장한 것이 **미들웨어**입니다. 클라이언트와 DBMS 사이에 **또 다른 서버**를 두는 방식입니다.

- 클라이언트는 중앙 서버에 **요청만** 보냅니다.
- 중앙 서버(미들웨어)가 **대부분의 로직을 수행**하고, 데이터 조작이 필요하면 DBMS에 부탁합니다.
- 덕분에 클라이언트는 화면 표시와 입력 전달만 담당해 매우 가벼워집니다.
- 로직이 바뀌어도 **중앙 미들웨어만 바꾸면** 됩니다.

## 웹 서버 vs WAS

최초의 웹은 정적 데이터만 보여줬지만, 동적 기능(입력·조회) 요구가 커지며 브라우저와 DBMS 사이의 미들웨어가 필요해졌습니다. WAS도 넓게 보면 미들웨어에 속합니다.

**웹 서버**는 정적 콘텐츠를, **WAS**는 동적 콘텐츠를 담당합니다.

| 구분 | 웹 서버 (Web Server) | WAS (Web Application Server) |
| --- | --- | --- |
| 처리 대상 | 정적 콘텐츠 (HTML·CSS·이미지) | 동적 콘텐츠 (DB 조회·비즈니스 로직) |
| 동작 방식 | HTTP로 바로 응답, 동적 요청은 WAS로 위임 | DB 서버와 함께 로직 처리 후 결과 반환 |
| 예시 | Apache HTTP Server, Nginx | Tomcat, Node.js, Jeus |

### 웹 컨테이너

JSP·서블릿을 실행시키는 소프트웨어를 웹 컨테이너(서블릿 컨테이너)라고 합니다. 웹 서버가 JSP를 요청하면 톰캣이 JSP를 서블릿으로 변환·컴파일해 수행 결과를 전달합니다. (Tomcat, Resin, WebLogic 등)

## 웹 서버와 WAS를 분리하는 이유

1. **역할 분담**: 정적 콘텐츠 요청은 웹 서버에 맡기고, WAS는 동적 콘텐츠에 집중해 서버 부담을 줄입니다.
2. **보안 강화**: WAS를 외부와 직접 연결하지 않아 리소스 노출 위험을 낮춥니다.
3. **로드 밸런싱**: 하나의 웹 서버에 여러 WAS를 연결해 요청을 분산합니다. 한 WAS에 장애가 나도 해당 WAS만 내리고 재시작해 **무중단 운영·장애 극복(Failover)** 이 가능합니다.

> **로드 밸런싱**: 서버가 처리할 요청(Load)을 여러 서버로 나누어(Balancing) 한쪽에 부하가 몰리지 않게 관리하는 것.

## 게이트웨이 (Gateway)

**한 네트워크에서 다른 네트워크로 이동하기 위해 거쳐야 하는 지점**입니다(네트워크의 톨게이트). 두 컴퓨터가 연결되려면 동일한 **통신 프로토콜**을 써야 하는데, 게이트웨이가 변환기 역할을 합니다.

- 게이트웨이 역할은 OSI 계층에 따라 나뉩니다. **1~3계층**은 주로 **라우터**가, **4~7계층**은 프로토콜 변환을 담당하는 소프트웨어 서버가 처리합니다.
- 인터넷 연결 시 IP 주소와 함께 **게이트웨이 주소**를 설정합니다.

### OSI 7계층 · TCP/IP

- **OSI 7 Layer**: 통신 과정을 7단계로 나눈 모델.
- **TCP/IP**: 네트워크 사이에서 정보를 주고받는 통신 프로토콜의 모음(인터넷 프로토콜 스위트).
- **통신 프로토콜**: 원활한 통신을 위해 약속한 규약(신호 처리, 오류 처리, 암호·인증, 주소 등). 예: HTTP, HTTPS(SSL 인증서 사용).

## URI와 URL

- **URI(Uniform Resource Identifier)**: 특정 리소스를 **식별**하는 통합 자원 식별자.
- **URL(Uniform Resource Locator)**: 리소스가 **어디 있는지** 알려주는 웹 주소. URI의 부분집합(subset)입니다.

---

정리하면, 웹 아키텍처의 발전은 "**로직을 어디에 둘 것인가**"의 역사입니다. 클라이언트 → 미들웨어 → 웹 서버/WAS 분리로 이어지며 확장성·보안·운영 안정성을 확보해 왔습니다.
