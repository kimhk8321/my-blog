성능 최적화는 "느린 곳을 찾아 측정하고, 고치고, 다시 측정하는" 반복 작업입니다. 감으로 고치기 전에 **무엇이 느린지 측정**하는 것이 먼저입니다. 이 글에서는 측정 도구부터 영역별 최적화 기법까지 순서대로 정리합니다.

## 무엇을 측정하는가 — Core Web Vitals

구글이 제시한 **Web Vitals**는 사용자 경험을 수치화한 지표입니다. 그중 핵심 3가지는 다음과 같습니다.

- **LCP (Largest Contentful Paint)**: 가장 큰 콘텐츠가 그려지는 시점. 로딩 체감 속도. (권장 2.5초 이내)
- **INP (Interaction to Next Paint)**: 사용자 입력에 대한 반응 지연. (권장 200ms 이내, 기존 FID를 대체)
- **CLS (Cumulative Layout Shift)**: 레이아웃이 밀리는 정도의 누적값. (권장 0.1 이하)

## 측정 도구

- **Lighthouse**: 크롬 개발자 도구에 내장. Web Vitals 기반으로 성능 점수와 개선 항목을 제공합니다.
- **Performance 탭**: 원하는 구간을 녹화하여 네트워크·렌더링·메모리 전반을 타임라인으로 확인합니다. 긴 작업(Long Task)과 리렌더링 병목을 찾을 때 유용합니다.
- **Memory 탭**: 현재 메모리 사용량과 스냅샷 간 차이를 비교해 **메모리 누수**가 발생하는 항목을 추적합니다.
- **Network 탭**: 각 요청의 처리 시간을 확인하고, 스로틀링 옵션으로 느린 모바일 네트워크를 시뮬레이션할 수 있습니다.

> 측정 없이 최적화하지 않는다 — 항상 **변경 전후를 같은 조건에서 비교**해 효과를 확인하세요.

## 렌더링 최적화

불필요한 리렌더링을 줄이는 것이 React 성능의 핵심입니다.

- **`React.memo`**: props가 변하지 않으면 컴포넌트 리렌더링을 건너뜁니다.
- **`useMemo`**: 비용이 큰 계산 결과를 메모이즈합니다.
- **`useCallback`**: 자식에게 내려주는 함수의 참조를 유지해 `memo`된 자식의 불필요한 렌더링을 막습니다.
- **리스트 가상화**: 수천 개의 행은 `react-window` / `@tanstack/virtual`로 화면에 보이는 항목만 렌더링합니다.
- **상태 위치 최적화**: 상태를 실제로 쓰는 컴포넌트에 가깝게 두어, 상태 변경의 렌더링 범위를 좁힙니다.

```tsx
const ExpensiveList = React.memo(function ExpensiveList({ items }: { items: Item[] }) {
  const sorted = useMemo(() => [...items].sort((a, b) => a.order - b.order), [items]);
  return (
    <ul>
      {sorted.map((it) => (
        <li key={it.id}>{it.label}</li>
      ))}
    </ul>
  );
});
```

> 주의: `memo`/`useMemo`/`useCallback`도 비용이 있습니다. **측정으로 병목이 확인된 곳**에만 적용하세요. 무분별한 메모이제이션은 오히려 코드 복잡도와 메모리만 늘립니다.

## 번들 최적화 — 코드 스플리팅

처음에 내려받는 JavaScript 양을 줄이면 초기 로딩이 빨라집니다.

- **코드 스플리팅**: `React.lazy`와 `Suspense`(또는 Next.js의 `dynamic`)로 필요한 시점에 컴포넌트를 불러옵니다.
- **트리 셰이킹**: 사용하지 않는 export를 번들에서 제거합니다. (ESM, `sideEffects: false` 설정)
- **번들 분석**: `@next/bundle-analyzer` 등으로 무거운 의존성을 찾아 더 가벼운 대안으로 교체하거나 지연 로딩합니다.

```tsx
import { lazy, Suspense } from "react";

const Chart = lazy(() => import("./Chart"));

function Dashboard() {
  return (
    <Suspense fallback={<div>차트 불러오는 중…</div>}>
      <Chart />
    </Suspense>
  );
}
```

> Vite 프로젝트라면 청크 분할(`manualChunks`)·압축 플러그인 등 **빌드 설정** 디테일은 [Vite + React 빌드 최적화](/posts/vite-build-optimization)에서 다룹니다.

## 네트워크 최적화

- **CDN**: 정적 자원을 사용자와 가까운 분산 서버에서 제공하여 거리를 최소화하고 로딩 속도를 높이며 원본 서버 부하를 줄입니다.
- **캐싱**: `Cache-Control`, ETag 등으로 브라우저·CDN 캐시를 활용합니다.
- **압축**: gzip/brotli로 전송 크기를 줄입니다.
- **요청 수 줄이기**: 번들·이미지 스프라이트·HTTP/2 멀티플렉싱을 활용합니다.

## 이미지·폰트 최적화

- **적절한 크기·포맷**: 화면에 필요한 해상도로 리사이즈하고 WebP/AVIF 같은 최신 포맷을 사용합니다.
- **지연 로딩**: 뷰포트 밖 이미지는 `loading="lazy"`로 나중에 불러옵니다.
- **Next.js `next/image`**: 자동 리사이즈·포맷 변환·지연 로딩을 제공합니다.
- **폰트**: `font-display: swap`으로 폰트 로딩 중 텍스트가 보이게 하고, 서브셋 폰트로 용량을 줄입니다.

## 정리

1. **측정** — Lighthouse·Performance 탭으로 병목을 찾는다.
2. **렌더링** — 메모이제이션과 가상화로 불필요한 렌더링을 줄인다.
3. **번들** — 코드 스플리팅·트리 셰이킹으로 초기 JS를 줄인다.
4. **네트워크·자원** — CDN·캐싱·이미지 최적화로 전송량을 줄인다.
5. **재측정** — 변경 효과를 같은 조건에서 확인한다.

성능 최적화는 한 번에 끝나는 작업이 아니라, 지표를 꾸준히 관찰하며 개선하는 **습관**에 가깝습니다.
