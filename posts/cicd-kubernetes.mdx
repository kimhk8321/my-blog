애플리케이션 개발 단계를 자동화하여, 더 짧은 주기로 고객에게 제공하는 방법인 CI/CD와 그 배포 기반인 컨테이너·쿠버네티스를 정리합니다.

## CI/CD

### 지속적 통합 (Continuous Integration, CI)

개발자는 하루에도 여러 번 코드를 푸시합니다. 저장소에 푸시할 때마다 **자동으로 빌드하고 테스트**하는 스크립트를 실행하면 오류 가능성을 줄일 수 있습니다. 제출된 각 변경사항이 모든 테스트·코드 표준을 통과하도록 보장하는 것이 CI입니다.

### 지속적 전달 (Continuous Delivery, CD)

CI를 넘어, 코드가 푸시될 때마다 빌드·테스트뿐 아니라 **배포 가능한 상태**까지 자동으로 준비합니다. 다만 실제 **배포는 사람이 수동으로(전략적으로) 트리거**합니다.

### 지속적 배포 (Continuous Deployment, CD)

지속적 전달과 유사하지만, 배포까지 **자동으로** 이루어져 사람의 개입이 필요 없습니다.

> **한 줄 요약**
> - **DevOps**: 개발(Development)과 운영(Operations)을 통합해 프로세스 속도를 높이는 접근.
> - **CI/CD**: 지속적으로 통합·테스트·배포하고, 이 흐름을 자동화하는 것.

통합·테스트부터 제공·배포까지 전체 라이프사이클에 걸친 자동화·모니터링 구축을 **"CI/CD 파이프라인"** 이라 부르며, 애자일·DevOps·SRE 방식으로 지원됩니다.

**대표 도구**: Git, Jenkins, GitLab CI, AWS CodeBuild, Azure DevOps 등.

### GitLab CI/CD 핵심 용어

- **Pipeline**: CI/CD의 최상위 구성 요소. Job과 Stage로 이루어집니다.
- **Job**: `.gitlab-ci.yml`의 가장 기본 요소. Runner가 실행하는 명령 모음.
- **Stage**: Job을 실행할 시기(순서)를 정의(예: build → test → deploy).
- **Artifacts**: Stage 간에 전달되는 결과물.
- **Cache**: 프로젝트 의존성을 저장해 재사용.
- **Runner**: `.gitlab-ci.yml`에 정의된 코드를 실제로 실행하는 주체.

> 📝 **업데이트**: 요즘은 **GitHub Actions**가 가장 널리 쓰이는 CI/CD 도구 중 하나입니다. 개념은 동일합니다 — 워크플로우(Pipeline) 안에 Job과 Step을 정의하고, 러너에서 실행합니다.
>
> ```yaml
> # .github/workflows/ci.yml
> name: CI
> on: [push]
> jobs:
>   build:
>     runs-on: ubuntu-latest
>     steps:
>       - uses: actions/checkout@v4
>       - run: npm ci
>       - run: npm test
> ```

## 컨테이너 · 도커 · 쿠버네티스

### 용어

- **컨테이너**: 앱이 구동되는 환경까지 감싸서 실행하는 **격리 기술**.
- **컨테이너 런타임**: 컨테이너를 다루는 도구 (예: 도커).
- **오케스트레이션**: 여러 서버에 걸친 컨테이너와 환경 설정을 관리하는 행위.
- **쿠버네티스**: 컨테이너 런타임을 통해 컨테이너를 오케스트레이션하는 도구.

### 컨테이너를 쓰는 이유

무거운 프로그램들은 설치 경로·옵션이 충돌하기 쉽습니다. 컨테이너는 프로그램 구동에 필요한 환경을 **격리**해, 여러 프로그램을 함께 돌려도 서로 간섭하지 않게 합니다(프로그램 사이에 장벽을 세우는 셈). 환경을 묶은 **컨테이너 이미지**를 내려받아 구동하면 별도 설정 과정이 줄어듭니다.

### 도커(컨테이너 런타임)

컨테이너를 쉽게 내려받고 공유·구동하게 해주는 가장 유명한 런타임입니다. 컨테이너 규격이 표준화되어 있어, 도커로 만든 컨테이너를 다른 런타임으로도 구동할 수 있습니다.

### 쿠버네티스(오케스트레이션)

도커 같은 런타임을 통해 **다수의 컨테이너 실행을 관리·조율**합니다. 컨테이너 하나를 돌릴 땐 도커면 충분하지만, 다수의 컨테이너를 관리·조율해야 할 때 쿠버네티스를 사용합니다. (자동 확장, 자가 복구, 롤링 업데이트 등)

## 배포 환경의 변천사

**전통적 배포 → 가상화 배포 → 컨테이너 중심 배포**

- **전통적 배포**: 한 컴퓨터에 하나의 OS를 깔고 여러 프로그램을 구동. 성능 저하·프로그램 충돌 문제.
- **가상화 배포**: 하이퍼바이저가 한 시스템에서 여러 가상 머신(VM)을 구동. 간섭을 막고 자원을 개별 할당할 수 있지만, **VM마다 OS를 설치**해야 해 무겁습니다.
- **컨테이너 중심 배포**: 하이퍼바이저 → 컨테이너 런타임, VM → 컨테이너로 대체. 컨테이너는 **하나의 OS를 공유**하므로 OS를 매번 설치할 필요가 없어 가볍습니다. 단, 컨테이너가 OS 환경에 문제를 일으키면 영향을 받을 수 있습니다.

---

정리하면, **CI/CD**는 "빌드·테스트·배포를 자동화하는 흐름"이고, **컨테이너·쿠버네티스**는 "그 결과물을 일관되고 확장 가능하게 배포·운영하는 기반"입니다.
