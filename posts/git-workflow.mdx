**Git**은 코드의 변경 이력을 관리하는 분산 버전 관리 시스템입니다. 기본 개념과 협업을 위한 브랜치 전략을 정리합니다.

## 핵심 개념

- **커밋(commit)**: 변경 사항의 스냅샷. 각 커밋은 이전 커밋을 가리켜 이력을 이룹니다.
- **브랜치(branch)**: 독립된 작업 흐름. 사실상 커밋을 가리키는 가벼운 포인터.
- **3가지 영역**: 작업 디렉터리 → (`add`) → 스테이징 → (`commit`) → 저장소.

```bash
git add .            # 변경을 스테이징
git commit -m "메시지" # 스냅샷 기록
git push origin main  # 원격에 반영
```

## 자주 쓰는 명령

```bash
git status            # 현재 상태
git log --oneline     # 이력 보기
git branch feature    # 브랜치 생성
git switch feature     # 브랜치 이동 (구: checkout)
git merge feature     # 병합
git rebase main       # 베이스를 옮겨 이력을 정리
git restore <file>     # 변경 취소
```

## merge vs rebase

- **merge**: 두 브랜치를 합치며 **병합 커밋**을 남김. 이력이 사실 그대로 보존됨.
- **rebase**: 커밋을 다른 베이스 위로 옮겨 **이력을 일직선**으로 정리. 깔끔하지만 이미 공유된(push된) 커밋엔 쓰지 않는 게 원칙.

## 브랜치 전략

팀 규모와 배포 주기에 따라 협업 방식을 정합니다.

### Git Flow

`main`/`develop`을 중심으로 `feature`/`release`/`hotfix` 브랜치를 둡니다. 복잡하지만 **정해진 릴리스 주기**가 있는 제품에 적합. 요즘 웹 서비스에는 다소 무거운 편.

### GitHub Flow

`main` 하나를 중심으로, 기능마다 브랜치를 따서 **PR(Pull Request) → 리뷰 → main 병합 → 배포**. 단순하고 **CI/CD·지속적 배포**와 잘 맞아 가장 널리 쓰입니다.

```
main ──●─────●──────●  (항상 배포 가능)
        \   /
   feature ●  → PR → 리뷰 → merge
```

### 트렁크 기반 개발 (Trunk-Based)

아주 짧은 수명의 브랜치(또는 main 직접)에 **자주, 작게** 통합. 강력한 테스트 자동화가 전제. 대규모 팀의 빠른 통합에 유리.

## 좋은 커밋 습관

- **작고 의미 있는 단위**로 커밋.
- 메시지는 "무엇을, 왜" 바꿨는지. (예: `fix: 로그인 시 토큰 갱신 누락 수정`)
- [CI/CD](/posts/cicd-kubernetes)와 연결하면, PR마다 자동 빌드·테스트로 품질을 지킬 수 있습니다.
