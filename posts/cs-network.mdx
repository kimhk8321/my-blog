## OSI 7계층과 TCP/IP

통신 과정을 계층으로 나눠 추상화한 모델입니다. 각 계층은 아래 계층의 서비스를 이용합니다.

| OSI 계층 | 예 | 역할 |
|---|---|---|
| 7 응용 | HTTP, DNS | 애플리케이션 프로토콜 |
| 4 전송 | TCP, UDP | 종단 간 전달, 신뢰성 |
| 3 네트워크 | IP | 주소 지정·라우팅 |
| 2 데이터링크 | Ethernet | 인접 노드 간 전송 |

실무에서는 더 단순한 **TCP/IP 4계층**(응용·전송·인터넷·네트워크 인터페이스)으로 이야기하는 경우가 많습니다.

## TCP vs UDP

전송 계층의 두 프로토콜입니다.

- **TCP**: **연결 지향**, 신뢰성 보장(순서·재전송·흐름 제어). 웹·파일 전송 등 정확성이 중요한 곳.
- **UDP**: **비연결**, 빠르지만 신뢰성 없음(유실 가능). 실시간 스트리밍·게임·DNS 등 속도가 중요한 곳.

## TCP 연결 — 3-way Handshake

TCP는 데이터 전송 전에 연결을 맺습니다.

```
클라이언트 → SYN →        서버
클라이언트 ← SYN+ACK ←   서버
클라이언트 → ACK →        서버   (연결 수립)
```

종료는 4-way handshake(FIN/ACK)로 이루어집니다. 이 연결 비용 때문에, 연결을 재사용하는 것이 성능에 중요합니다.

## HTTP의 진화

- **HTTP/1.1**: 요청-응답 1:1. **HOL Blocking**(앞 요청이 막히면 뒤도 대기) 문제. Keep-Alive로 연결은 재사용.
- **HTTP/2**: 하나의 연결에서 **멀티플렉싱**(여러 요청 병렬 처리), 헤더 압축, 서버 푸시. HOL Blocking을 애플리케이션 계층에서 해소.
- **HTTP/3**: 전송을 TCP가 아닌 **QUIC(UDP 기반)** 으로. TCP의 HOL Blocking까지 제거하고 연결 수립이 빨라짐.

> "요청을 어떻게 더 많이, 더 빨리, 막힘 없이 보내는가"의 역사입니다.

## 그 외 핵심

- **DNS**: 도메인(`example.com`)을 IP로 변환하는 "전화번호부".
- **HTTPS**: HTTP + **TLS** 암호화. 데이터 도청·변조를 막고 서버 신원을 인증.
- **로드 밸런싱**: 여러 서버로 트래픽 분산 ([웹 서버 아키텍처](/posts/web-server-architecture) 참고).
