> 이 글은 비어 있던 노트를 새로 정리한 것입니다. "어디서, 언제 HTML을 만드는가"라는 관점에서 렌더링 전략을 비교합니다.

## CSR (Client-Side Rendering)

브라우저가 거의 빈 HTML과 JavaScript 번들을 받아, **클라이언트에서** 자바스크립트를 실행해 화면을 그립니다. (전형적인 SPA)

- 👍 페이지 전환이 빠르고, 서버 부하가 적습니다.
- 👎 초기 로딩 시 빈 화면(번들 다운로드·실행 대기)이 있고, 검색 엔진/링크 미리보기에 불리할 수 있습니다.

## SSR (Server-Side Rendering)

요청이 올 때마다 **서버에서** HTML을 완성해 내려줍니다. 브라우저는 바로 내용을 볼 수 있고, 이후 자바스크립트가 로드되면 상호작용이 가능해집니다.

- 👍 빠른 첫 화면(FCP)과 좋은 SEO. 매 요청마다 최신 데이터를 반영.
- 👎 요청마다 서버가 렌더링하므로 서버 부하와 TTFB가 늘 수 있습니다.

### 하이드레이션(Hydration)

SSR이 내려준 정적인 HTML에 React가 **이벤트 핸들러와 상태를 연결**해 살아 있는 페이지로 만드는 과정입니다. 서버 렌더 결과와 클라이언트 첫 렌더 결과가 일치해야 하며, 다르면 "hydration mismatch" 경고가 발생합니다.

```
서버에서 HTML 생성 → 브라우저에 표시(아직 정적) → JS 로드 → 하이드레이션 → 상호작용 가능
```

## SSG (Static Site Generation)

**빌드 시점에** 페이지를 미리 HTML로 생성해 둡니다. 요청이 오면 만들어진 정적 파일을 그대로 제공합니다(CDN 캐싱에 유리).

- 👍 가장 빠르고 저렴하며 안정적입니다. 블로그·문서처럼 자주 바뀌지 않는 콘텐츠에 적합.
- 👎 데이터가 바뀌면 다시 빌드해야 합니다.

## ISR (Incremental Static Regeneration)

SSG의 장점을 유지하면서, **일정 주기 또는 요청에 따라** 백그라운드에서 정적 페이지를 다시 생성합니다. "정적의 속도 + 어느 정도의 최신성"을 절충한 방식입니다(Next.js).

```ts
// Next.js: 60초마다 백그라운드 재생성
export const revalidate = 60;
```

## 어떻게 고를까

| 전략 | 생성 시점 | 적합한 경우 |
|---|---|---|
| **CSR** | 클라이언트 | 로그인 후 대시보드 등 SEO가 덜 중요한 앱 |
| **SSR** | 요청 시(서버) | 매번 최신·개인화 데이터가 필요하고 SEO가 중요 |
| **SSG** | 빌드 시 | 블로그·문서 등 거의 바뀌지 않는 콘텐츠 |
| **ISR** | 빌드 + 주기적 갱신 | 정적이되 가끔 갱신되는 콘텐츠(상품 목록 등) |

> 📝 **참고 (Next.js App Router)**: 최근 Next.js는 **React Server Components(RSC)** 기반으로, 컴포넌트 단위로 서버/클라이언트 렌더링을 섞습니다. 위 전략들은 여전히 유효한 사고틀이지만, 이제는 페이지 전체가 아니라 **컴포넌트별로** 렌더링 위치를 정할 수 있다는 점이 핵심 변화입니다.

---

정리하면, 렌더링 전략 선택은 결국 **"최신성 · 속도 · 서버 비용 · SEO"** 사이의 트레이드오프입니다. 콘텐츠의 성격에 맞춰 고르면 됩니다.
